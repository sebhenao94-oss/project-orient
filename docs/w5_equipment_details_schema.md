## `public.equipment_details` (12 columns)

| column | type | nullable | notes |
|---|---|---|---|
| `equipment_id` | bigint | NO | `GENERATED BY DEFAULT AS IDENTITY` — never hand-allocated; INSERT omits it |
| `name` | text | YES | encodes the unit **and its type** via `{Type}_{floor}-{unit}` (e.g. `AHU_2-01`) |
| `property_id` | uuid | YES | building / property id |
| `floorRef` | text | YES | bare floor digit (`"2"`); null for building-level plants |
| `zone` | text | YES | |
| `spaceRef` | text | YES | |
| `systemRef` | bigint | YES | self-ref to a parent equipment row (a system/plant) |
| `systemRef_type` | text | YES | describes the **systemRef parent relationship** — NOT the equipment's own type |
| `airRef` | bigint | YES | → parent `equipment_id` (ATU → AHU) |
| `chilledWaterRef` | bigint | YES | → plant `equipment_id` |
| `hotWaterRef` | bigint | YES | → plant `equipment_id` |
| `condenserWaterRef` | bigint | YES | → plant `equipment_id` |

All camelCase identifiers must be **double-quoted** in SQL (Postgres folds
unquoted names to lowercase). Only `equipment_id` is `NOT NULL`, and it is the
identity column, so an INSERT of the other fields cannot fail on a missing
required column.

## Relationships: ref-columns, not a separate table

There is **no `equipment_relationships` table** in `public` (confirmed empty
lookup). Equipment-to-equipment relationships are stored as the bigint ref
columns above on the **child's** row (e.g. `child.airRef = parent.equipment_id`).
Each child therefore holds at most one ref of each kind — consistent with the
graph validator's "one airRef per terminal" rule.

## Implications for A4 (applied in `review_store.py`)

- INSERT/UPDATE write `name, property_id, floorRef, zone, spaceRef` and let the
  identity column assign `equipment_id`. **There is no type column**, so the type
  is carried by `name`; we do **not** write `equipment_type` into `systemRef_type`
  (corrected from the initial A4 draft — `systemRef_type` is reserved for a real
  systemRef parent).
- Relationship actions UPDATE the matching bigint ref column on the child row.
- **No sequence grant is required** (identity column; `orient_team_4` already has
  `INSERT` on `equipment_details`).
